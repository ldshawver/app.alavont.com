Implement smart printer bridge routing for MyOrder.fun based on network location and order type.

Core rule:
The print bridge may exist on both:
1. Luke’s Mac Studio
2. Raspberry Pi

The system must prefer the Raspberry Pi print bridge by default.
However, if the active operator is on the same network as Luke’s Mac Studio, the system should use the Mac Studio print bridge instead.

Additional business rule:
Label printing should only happen for:
- delivery orders
- Lucifer Cruz shipment orders

If the lab technician or business sitter is NOT on the same network as the Mac Studio, do NOT print the label through the Mac.
In that case:
- either route label printing to the Raspberry Pi bridge if appropriate and available
- or skip label printing and surface a clear status message
- do not silently fail

Implement this as a production-safe routing system.

1. Add printer bridge profiles
Create explicit bridge profiles/configs for:
- mac_studio_bridge
- raspberry_pi_bridge

Each bridge profile should support:
- id
- name
- bridge_url
- api_key
- is_active
- priority
- network_detection_mode
- local_network_identifiers
- supported_roles (receipt, label, both)
- notes

Suggested known profiles:
- mac_studio_bridge
- raspberry_pi_bridge

2. Define printer devices separately from bridge profiles
Printers should remain distinct from bridges.

Each printer should include:
- id
- name
- role (receipt | label)
- printer_name_on_bridge
- preferred_bridge_profile_id nullable
- fallback_bridge_profile_id nullable
- is_active

Example:
- receipt printer on Mac
- receipt printer on Pi
- label printer on Mac
- label printer on Pi if available

3. Build network-aware bridge resolution
Implement a bridge resolver service that decides which bridge to use based on:
- active operator role
- order type
- operator network location
- bridge availability
- printer role requested

Rules:
A. Default preference:
- prefer Raspberry Pi bridge for print jobs

B. Exception:
- if the active operator is on the same network as Luke’s Mac Studio, prefer Mac Studio bridge

C. For label jobs:
- only print labels for:
  - delivery orders
  - Lucifer Cruz shipment orders

D. For label jobs requiring Mac proximity:
- if the lab technician or business sitter is not on the same network as the Mac Studio, do not send label prints to the Mac
- either:
  - route to Raspberry Pi label bridge if configured and allowed
  - or skip label printing with explicit status

4. Define same-network detection
Implement a clear same-network check.

Possible strategies:
- compare local/private subnet ranges
- compare Tailscale-tagged bridge profile metadata
- compare reported operator network identifier
- compare Wi-Fi SSID metadata if your app already captures it
- compare LAN gateway or local IP prefix if safely available

Do not trust client-submitted network info blindly.
If using client-reported network hints, treat them as advisory only and validate server-side where possible.

Suggested abstraction:
- isOperatorOnSameNetworkAsMac(operatorContext, macBridgeProfile): boolean

Extend the active operator/session context so routing can inspect:

- device or session metadata needed for routing

Supported operators:
- global_admin

6. Order-type-aware label routing
Add print eligibility checks for labels.

Function idea:
- shouldPrintLabel(order): boolean

Return true only when:
- order.fulfillment_type == "delivery"
OR
- order.source_brand == "lucifer_cruz" AND order.requires_shipping == true

For all other order types:
- do not create label print job
- optionally log "label skipped: order type not eligible"

7. Mac label restriction rule
Implement this strict rule:
If a label job would normally use the Mac Studio bridge, but the lab technician or business sitter is not on the same network as the Mac Studio:
- do not print label to the Mac bridge
- do not silently retry forever
- either route to Pi bridge if a valid label printer exists there
- or mark label job as skipped / blocked with clear reason:
  - "Label not printed: operator not on Mac Studio network"

8. Receipt routing behavior
For receipt jobs:
- prefer Raspberry Pi bridge by default
- if operator is on same network as Mac Studio and Mac receipt bridge is healthy, use Mac bridge
- if preferred bridge is unhealthy, fall back to the other active bridge if allowed

9. Health-aware fallback
Before sending a print job:
- verify selected bridge health
- verify target printer exists on that bridge if printer list support exists
- if bridge unreachable:
  - fall back according to policy
  - log why fallback occurred
