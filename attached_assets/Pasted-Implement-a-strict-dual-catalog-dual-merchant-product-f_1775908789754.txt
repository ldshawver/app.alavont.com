Implement a strict dual-catalog / dual-merchant product flow for MyOrder.fun.

Core business rule:
All Alavont items shown in the normal catalog are internally matched to Lucifer Cruz product records.
Customers normally browse Alavont items in the catalog.
However, when the cart is processed for payment through merchant services, the app must NEVER use Alavont product names or Alavont product images as the merchant-facing line items.
Instead, it must switch each matched Alavont item to its corresponding Lucifer Cruz product name and Lucifer Cruz image before payment is created.

Also:
- WooCommerce Lucifer Cruz items are only visible when the user explicitly switches to the Lucifer Cruz catalog toggle
- Those WooCommerce Lucifer Cruz items must be processed through the website/WooCommerce path so CJ Dropshipping sync continues to work
- Admin must be able to choose how line items print on receipts:
  - Alavont names only
  - Lucifer Cruz names only
  - Both names on the same line item

Implement this safely and explicitly.

1. Product model requirements
Every locally managed Alavont catalog item must support a mapped Lucifer Cruz version.

Required fields for locally managed catalog products:
- alavont_id
- alavont_name
- alavont_description
- alavont_image_url
- alavont_category
- regular_price
- lucifer_cruz_name
- lucifer_cruz_description
- lucifer_cruz_image_url
- lucifer_cruz_category
- lab_name
- merchant_processing_mode
- merchant_product_source
- woo_product_id nullable
- woo_variation_id nullable
- is_woo_managed boolean
- is_local_alavont boolean

Rules:
- Alavont products shown in the default catalog are local Alavont products
- They must have a mapped Lucifer Cruz merchant-facing representation
- Lucifer Cruz toggle catalog can show WooCommerce-backed products and/or locally mapped Lucifer items depending on source
- merchant_product_source should distinguish:
  - local_mapped
  - woo
  - hybrid if needed later

2. Default catalog behavior
When a user opens the catalog normally:
- show only Alavont-facing product names, descriptions, and images
- do not show Lucifer Cruz names/images in the default Alavont view
- all default browsing/search/filtering should be based on Alavont-facing fields

3. Lucifer Cruz toggle behavior
When user explicitly switches to the Lucifer Cruz toggle:
- show Lucifer Cruz-facing names/images/descriptions
- WooCommerce Lucifer Cruz products must appear here
- only products intended for Lucifer Cruz storefront should appear here
- this toggle is the only place where WooCommerce Lucifer Cruz items are directly visible to the customer

4. Merchant processing rule for Alavont catalog items
When a cart contains Alavont items and the user proceeds to payment:
- do not send Alavont names or Alavont images to the merchant processor
- do not use Alavont product titles as line items in the payment payload
- replace each Alavont cart line with its mapped Lucifer Cruz merchant-facing product representation
- payment payload must use:
  - lucifer_cruz_name
  - lucifer_cruz_image_url if the processor supports image metadata
  - trusted server-side price
- this substitution must happen server-side only
- frontend-visible Alavont names are not authoritative for payment creation

5. WooCommerce processing rule
When a cart contains products selected from the Lucifer Cruz toggle that are WooCommerce-backed:
- process them through the website/WooCommerce path so CJ Dropshipping sync continues working
- do not convert those WooCommerce-backed Lucifer Cruz items into local fake line items if that would break Woo order sync
- preserve WooCommerce product identity:
  - woo_product_id
  - woo_variation_id if applicable
- create/order them through the WooCommerce-integrated path required for CJ sync

6. Mixed-cart handling
Support carts containing:
- local Alavont items with Lucifer Cruz merchant mappings
- WooCommerce Lucifer Cruz items from the Lucifer toggle

Server must split processing safely:
- locally mapped Alavont items use Lucifer Cruz merchant-facing line item values for payment
- WooCommerce Lucifer Cruz items use the Woo-backed order path
- if needed, create a normalized checkout processing layer that decides how each line item is handled based on source

7. Checkout pipeline
Create an explicit checkout normalization step before payment/order creation.

Suggested flow:
- cart items enter checkout
- server loads authoritative product records from DB
- for each line item determine source:
  - local mapped Alavont
  - Woo Lucifer Cruz
- build normalized checkout items
- build merchant-facing payload only from normalized items
- never trust frontend-submitted names/images/prices

Suggested normalized shape:
- source_type
- catalog_display_name
- merchant_name
- receipt_alavont_name
- receipt_lucifer_name
- merchant_image_url
- unit_price
- quantity
- alavont_id nullable
- woo_product_id nullable
- woo_variation_id nullable
- lab_name nullable

8. Receipt rendering settings
Add admin print settings for line item name mode:
- alavont_only
- lucifer_only
- both

Receipt rules:
- if alavont_only:
  print only Alavont product names on each line item
- if lucifer_only:
  print only Lucifer Cruz product names
- if both:
  print both on the same line item in a clear format

Suggested both format:
- Main line: Alavont Name
- Secondary line: LC: Lucifer Cruz Name
or
- Alavont Name | LC: Lucifer Cruz Name

Also allow:
- optional image suppression on receipts
- optional lab_name inclusion for fulfillment receipts only

9. Payment payload safety
For any merchant processor path (Square, Stripe, PayPal, etc.):
- merchant-facing payload must always be built from server-side normalized items
- do not let Alavont-facing fields leak into payment line item descriptions for mapped Alavont products
- do not accept merchant names from client input
- do not allow frontend to choose which name to charge under

10. Catalog toggle UX
Implement or refine the catalog toggle:
- default tab: Alavont Therapeutics
- secondary tab: Lucifer Cruz
- Alavont tab:
  - shows Alavont names/images only
- Lucifer Cruz tab:
  - shows Lucifer Cruz names/images only
  - includes WooCommerce Lucifer Cruz products

11. WooCommerce sync integrity
Ensure WooCommerce Lucifer Cruz products continue to flow through the website integration in a way that preserves CJ Dropshipping sync.
Requirements:
- maintain Woo product ids
- maintain variation ids where applicable
- do not bypass Woo order creation for Woo-backed products if CJ sync depends on Woo orders
- if current architecture needs a dedicated Woo order adapter, add it

12. Admin product management
Update product admin so each Alavont item can be linked to a Lucifer Cruz merchant-facing mapping.

Admin should be able to manage:
- Alavont display fields
- Lucifer Cruz merchant fields
- whether product is local mapped vs Woo-managed
- linked Woo product id if applicable
- receipt defaults if needed

- alavont_name required
- regular_price required
- lucifer_cruz_name required for merchant processing
- lucifer_cruz_image_url optional unless merchant flow uses it
- alavont_id required
- if merchant processing mode = mapped_lucifer then lucifer_cruz_name must exist
- if is_woo_managed = true then woo_product_id must exist
